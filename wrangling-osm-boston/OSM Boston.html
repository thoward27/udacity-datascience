def sq_ft_to_sq_meters(cell):
    """Small function to parse sq ft and replace with unformatted sq meters
    
    Pass in cell, using .apply method of Pandas onto desired vector.
    """
    match = re.search(regex_sq_ft, cell)
    target = match['number']
    num = target.replace(',','')
    new = int(num) * 0.092903
    new = (f"{new:2.2f}")
    cell = cell.replace(target, new)
    cell = cell.replace(match['units'], "sq meters")
    return cell
    
tags_with_feet['v'] = tags_with_feet['v'].apply(sq_ft_to_sq_meters)

Cuisine Tagged Nodes with Compound Types¶

Another problem I noted, was the compound types of cuisine contained within certain nodes. Since technically this is a violation of third normal form, I decided these values should be parsed out and forced into a transactions table.

tblRestaurant_cuisine_types¶

To properly build the table:

sql_insert = "INSERT OR IGNORE INTO tblRestaurant_cuisine_types (n_id, v) VALUES ({0}, {1})"
for node in places_to_eat[places_to_eat.k == "cuisine"]:
    v = node['v']
    if ";" in v: arr = v.split(";")
    if "," in v: arr = v.split(",")
    if len(arr) > 0:
        for v in arr:
            c.execute(sql_insert.format(node['n_id'], v)
        return True
    c.execute(sql_insert.format(node['n_id'], v)
    return True

Final thoughts on Cuisine Types¶

The combination of n_id and v should provide the primary key, as a restaurant should not be tagged American / Italian / Sandwiches / etc., more than a single time.

def generate_mp(loc):
    """Generates missing person for given index"""
    m_p = users.iloc[1].copy()
    m_p = users.sum(axis=0, numeric_only=True)
    m_p[m_p > 0] = 0
    m_p['u_name'] = "missing person"
    users.loc[loc] = m_p
    return True

def parse_uid(u_id):
    """Parses u_id and returns u_name, generates mp if no user exists"""
    try:
        return users.loc[u_id]['u_name']
    except Exception as err:
        generate_mp(u_id)
    try:
        return users.loc[u_id]['u_name']
    except Exception as err:

After applying these functions I reviewed my findings, 831 usernames were missing.

At this point alarm bells are going off in my head, and I posed a possible solution:

So there are 831 users who apparently have no corresponding u_name in the users table. This is quite odd and has rather few explinations. Especially since my parsing script printed no errors during the parse. This leads me to believe the error may be a genuine one inside of the database. The best explination I can conjure is that the nodes had u_ids without corresponding u_names; this could be because the user accounts were deleted after posting the node. The node may rely on a the u_id, or the delete script may not strip the u_id from each node, but may strip u_name in order to free that u_name to be re-registered by another user. With the help of another hidden u_transactions table, the DB Admin could theoretically go backwards in time and see which u_name posted each node historically, whether the u_name had changed hands 100 times or none. This type of asymetric cleansing of the data for deleted user accounts should be addressed in later analysis, as it could be a very efficient way to allow u_names to be fluid, yet maintain a record of who did what.

Coming back home¶

After this mind-blowing answer, I soon realized I had not been working on a solid base. I started by first pulling the entire table, without counts into a DataFrame, then reapplying my two functions. Which worked flawlessly and made no missing persons :)

Improving Boston¶

I think one way to improve the data in Boston is to gamify the process. The way I imagine it would work is, as you explore, and uncover more of the map, you're able to find weapons, tools, batteries, supplies, etc.. If you map something totally new, the odds of the reward being first tier are much higher. Then, allow users to battle eachother in the wild, envision it as almost a live-action Fall Out style game. From there publish a leader board, those who map the most would level the highest, thus creating a positive feedback loop towards contributions.

An advantage to this approach is that it is semi-hands-off once in production, drawbacks are obviously production costs. It's possible that these production costs could be offset, if another company were to want to create the game and sponser it with ads OSM would only have to provide their mapping data.
